## 🎯 Resumo executivo

Esta release entrega o **primeiro fluxo completo de entrada do usuário no Mintly** — da criação da conta ao acesso seguro e à recuperação de senha:

1. **`mintly-lib` v1.2.0** — os contratos de dados do cadastro e da autenticação (usuário, restaurante, contas e categorias financeiras) passam a morar na biblioteca compartilhada, com a **política de senha unificada** e os clients HTTP prontos para o frontend consumir.
2. **`MintlyApi`** — entrega as funcionalidades **MIN-58 (Cadastro de Usuário e Restaurante)** e **MIN-59 (Autenticação e Recuperação de Acesso)**, com onboarding automático do restaurante e segurança de acesso de ponta a ponta.

Diferente da release anterior (fundação técnica), **esta release tem funcionalidade visível para o usuário final**: criar conta, fazer login, recuperar a senha por e-mail e sair com segurança.

---

## 💡 Por que isso importa para o produto

| Antes | Depois |
|-------|--------|
| Sem porta de entrada: nenhum usuário conseguia criar conta | Cadastro completo com restaurante e configuração inicial automáticos |
| Sem controle de acesso real nas rotas | Login com sessão segura (JWT criptografado), logout e expiração |
| Senha esquecida = conta perdida | Recuperação por e-mail com link seguro de uso único (válido por 1h) |
| Sem rastro de quem acessou o quê | Auditoria de login, logout, falhas, bloqueios e redefinições — por restaurante |
| Tentativas ilimitadas de senha (porta aberta a ataques) | Bloqueio temporário após 5 tentativas inválidas |

**Benefícios:**
- ✅ **Onboarding sem fricção** — ao criar a conta, o sistema já cria o restaurante, a conta financeira padrão e as categorias de receitas/despesas típicas de um restaurante (Venda Balcão, Delivery, CMV/Insumos, Salários, Aluguel, Impostos).
- ✅ **Segurança de verdade** — senhas com hash forte, mensagens que não revelam se um e-mail existe (anti-enumeração), bloqueio de força bruta e invalidação de todas as sessões ao redefinir a senha.
- ✅ **Multi-tenant pronto** — toda requisição autenticada já carrega o restaurante do usuário extraído do token; as próximas features (contas, movimentações) só precisam filtrar por ele.
- ✅ **Confiança auditável** — trilha completa de eventos de acesso para suporte e conformidade.

## 📋 O que foi entregue

### Biblioteca — `mintly-lib` v1.2.0
1. Contratos do domínio de cadastro/autenticação: **Usuário** (credencial separada dos dados pessoais), **Restaurante**, **Conta Financeira** e **Categoria Financeira**, com schemas de validação Sapphire.
2. **Política de senha única** compartilhada entre cadastro e redefinição (mínimo 8 caracteres, 1 maiúscula, 1 minúscula, 1 número).
3. Clients HTTP (`AuthClient`, `RestaurantClient`, `FinancialAccountClient`, `FinancialCategoryClient`) para o frontend consumir a API sem reescrever contratos.
4. Atualização de versão: **1.1.0 → 1.2.0** (publicada no npm; promoção do canal `next` → `latest` na conclusão da release).

### API — `MintlyApi` (release `release-20260610`)
1. **Cadastro (MIN-58)** — signup cria usuário + restaurante + vínculo + dados iniciais numa única transação; e-mail único garantido por índice no banco.
2. **Login e sessão (MIN-59)** — JWT criptografado com refresh token rotacionado; registro de IP/dispositivo e último acesso.
3. **Proteções de acesso** — bloqueio temporário após 5 tentativas inválidas (15min); contas inativas/bloqueadas não autenticam; respostas genéricas que não revelam se o e-mail existe.
4. **Recuperação de senha** — link por e-mail (Resend) com token de uso único e validade de 1h; no banco fica apenas o hash do token; redefinir a senha derruba todas as sessões anteriores.
5. **Logout seguro** — exige autenticação, revoga o refresh token e audita.
6. **Auditoria** — eventos de cadastro, login, falhas, bloqueios, recuperação e redefinição gravados com o restaurante do usuário.
7. **Fundação de testes** — suíte com cobertura mínima de 90% (193 testes) + verificação manual de todas as rotas contra o servidor real (70 verificações de sucesso e erro, todas aprovadas).
8. **Ferramentas para o time** — collection do Insomnia com todas as rotas (`insomnia-mintly-api.json`) e script de verificação manual reutilizável (`scripts/manual-route-check.mjs`).
9. A **versão da API permanece `1.0.0`** — o marco é registrado pela tag/release **`release-20260610`** no GitHub.

## 🔒 Riscos e impacto

- **Risco:** Médio-baixo. Funcionalidade nova de autenticação — não altera contratos existentes, mas passa a proteger a rota de pessoas (`/people`) com token (comportamento já presente em staging desde a MIN-58).
- **Impacto em produção:** O frontend precisa enviar o token `Bearer` nas rotas protegidas e consumir `mintly-lib@1.2.0`.
- **Configuração nova exigida no ambiente:** `RESEND_API_KEY`, `EMAIL_FROM`, `FRONTEND_URL` (e-mail de recuperação); opcionais com defaults: `MAX_LOGIN_ATTEMPTS`, `BLOCK_DURATION_MINUTES`, `RECOVERY_TOKEN_EXPIRY_HOURS`.
- **Validação realizada:**
  - Verificação de tipos e lint ✔️ (lib e API)
  - Build da biblioteca ✔️
  - Testes automatizados da API: 193/193, cobertura 96,7% ✔️
  - Verificação manual de todas as rotas contra servidor real + MongoDB Atlas: 70/70 ✔️
  - Testado no ambiente de **desenvolvimento (`staging`)** — lib via PRs #6 e #7; API via PRs #11 e #12 ✔️
  - Validação final via CI no merge para produção (`main`) — em andamento.
- **Observação conhecida (não bloqueante):** buscar uma pessoa com id em formato inválido devolve erro 500 genérico em vez de 400 — registrado para correção no próximo ciclo.
